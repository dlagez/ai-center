# 单体服务目录结构说明

本文档用于说明 AI 中台项目当前采用的目录结构设计。

结论：

- 使用单一服务架构，不采用微服务拆分。
- 使用 `FastAPI + LangChain + LangGraph + LangSmith` 作为核心技术方向。
- 通过目录分层和模块分域的方式管理复杂度，而不是通过多服务拆分系统。

## 2. 设计原则

### 2.1 单体服务，不做微服务拆分

当前阶段优先保证：

- 开发效率
- 功能集中落地
- 调试链路清晰
- 部署复杂度可控

因此系统采用单一服务架构，在一个代码仓库、一个服务进程内承载核心能力。

### 2.2 按“分层 + 分域”组织代码

为了避免单体项目后期失控，目录设计采用两种维度同时约束：

- 分层：`api`、`governance`、`runtime`、`repositories`、`services` 等负责不同技术职责。
- 分域：`modules` 下按模型、Prompt、Agent、知识库、评测、发布等业务中心拆分。

这样可以在不拆服务的前提下，保留清晰的演进边界。

### 2.3 接口层只做入口，不承载核心业务

`app/api/` 只负责：

- 路由注册
- 请求校验
- 响应封装
- 身份上下文传递

具体业务逻辑应下沉到：

- `app/services/`
- `app/modules/`
- `app/runtime/`

### 2.4 运行时能力集中沉淀

与大模型运行相关的共性能力统一收敛到 `app/runtime/`，包括：

- 模型接入与路由
- fallback
- Tool Calling
- 工作流编排
- 检索与 RAG
- 记忆与状态管理

这样可以避免各业务中心重复实现一套模型调用逻辑。

## 3. 与架构图的映射关系

- AI 中台门户层、应用接入与开放层：映射到 `app/api/`
- 网关与治理层：映射到 `app/governance/`
- Agent 编排与运行层：映射到 `app/runtime/workflows/` 与 `app/modules/agent_center/`
- Prompt 工程层：映射到 `app/modules/prompt_center/`
- 模型与工具抽象层：映射到 `app/runtime/` 与 `app/integrations/model_providers/`
- 知识库与检索层：映射到 `app/modules/knowledge_center/` 与 `app/runtime/retrieval/`
- 记忆与状态层：映射到 `app/runtime/memory/`
- 观测、评测、部署层：映射到 `app/observability/` 与 `deploy/`
