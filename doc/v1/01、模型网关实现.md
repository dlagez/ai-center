# 模型网关实现 PRD（V1）

## 1. 文档目的

本文档用于明确 AI 中台在 V1 阶段如何落地“模型网关”能力，并把产品目标、职责边界、实现方案、接口约定、配置要求、数据模型和验收标准写清楚。

本文档解决的问题不是“是否要用 LiteLLM Proxy”，而是“在本项目里，如何把 LiteLLM Proxy 用成一个真正可落地的模型网关”。

---

## 2. 背景

当前项目已经明确以下方向：

- 公有模型以 LiteLLM Proxy 作为主网关。
- 私有模型、本地模型、内网模型保留例外直连能力。
- 上层业务不直接依赖 OpenAI、Anthropic、Gemini 等 provider SDK。
- `model_center` 负责目录、策略和配置。
- `runtime/llm` 负责统一调用编排和执行分发。
- `integrations/model_providers` 负责 LiteLLM Proxy 与直连例外 provider 的协议适配。

如果没有模型网关，后续会出现以下问题：

- 不同模块分别维护 OpenAI、Anthropic、Gemini 的调用代码，接口和错误处理不统一。
- 路由、fallback、限流、成本统计散落在业务代码里，后续无法集中治理。
- Prompt、Agent、知识库模块各自绑定具体模型，模型替换成本高。
- 多租户、多应用、多环境下的模型权限和成本边界无法稳定控制。

因此，V1 需要先把“统一模型网关”建起来，作为后续 Prompt、Agent、RAG、评测体系的共同底座。

---

## 3. V1 目标

### 3.1 产品目标

- 提供统一的模型调用入口，屏蔽公有模型 provider 差异。
- 让公有模型默认通过 LiteLLM Proxy 访问，形成单一主网关。
- 保留私有模型和高合规场景的例外直连能力。
- 支持基于租户、应用、任务类型、能力要求的逻辑模型选择。
- 支持稳定的 fallback 机制，但避免应用和网关双重 fallback。
- 记录 token、cost、latency、错误信息，形成基础观测闭环。

### 3.2 业务价值

- 降低多模型接入复杂度。
- 提高模型替换、切换、灰度和下线的效率。
- 为后续成本治理、模型评测、路由优化打基础。
- 为 LangGraph、Prompt 管理、知识库中心提供统一底层接口。

---

## 4. V1 非目标

以下内容不属于本次 V1 必做范围：

- 不做完整的模型管理后台 UI。
- 不做自动学习型动态路由。
- 不做精确到财务结算级别的账单系统。
- 不做图片、音频、视频等多模态完整能力闭环。
- 不做 embedding、rerank、responses API 的完整产品化支持。
- 不做每个租户一套独立 LiteLLM virtual key 的自动化生命周期管理。

说明：

V1 只把最核心的“公有模型主网关 + 例外直连 + 策略路由 + 统一调用 + 基础观测”先跑通。

---

## 5. 术语说明

### 5.1 上游 provider

指实际提供模型能力的服务商，例如：

- OpenAI
- Anthropic
- Gemini
- Azure OpenAI
- 本地模型
- vLLM

### 5.2 接入通道

指应用如何访问模型能力：

- `litellm_proxy`
- `direct`
- `internal_proxy`

### 5.3 逻辑模型

面向业务暴露的稳定模型标识，不直接等于 provider 原始模型名。  
例如：

- `chat_default`
- `chat_low_cost`
- `code_primary`
- `private_sensitive`

### 5.4 网关模型名

LiteLLM Proxy 配置里暴露出来的 `model_name`，供应用统一调用。  
例如：

- `public-chat-default`
- `public-code-default`

---

## 6. 方案结论

V1 采用以下统一方案：

1. 公有模型默认全部走 LiteLLM Proxy。
2. 应用侧只持有模型网关配置，不默认持有全部公网 provider 凭据。
3. `model_center` 维护逻辑模型目录、策略和权限，不直接发起 provider 请求。
4. `runtime/llm` 负责把逻辑模型解析成“LiteLLM Proxy 模型名”或“直连例外目标”，然后发起统一调用。
5. `integrations/model_providers` 提供 LiteLLM Proxy 适配器和私有模型直连适配器。
6. 公有模型的跨 provider 路由与 fallback 默认由 LiteLLM Proxy 执行。
7. 应用侧只允许做“通道级兜底”，不允许和 LiteLLM Proxy 做同级 fallback。

换句话说：

- 模型选择策略在应用侧。
- 公有模型执行路由在 LiteLLM Proxy 侧。
- 例外直连执行在应用侧 provider adapter。

## 7. 用户与使用场景

### 7.1 管理员场景

- 作为平台管理员，我希望配置逻辑模型和网关模型映射，以便业务不感知底层 provider 变化。
- 作为平台管理员，我希望按租户、应用控制模型可见性和可用性。
- 作为平台管理员，我希望查看每次模型调用的耗时、token、成本和失败原因。

### 7.2 业务模块场景

- 作为 Prompt 模块，我希望只传入逻辑模型或场景标识，而不是硬编码 `gpt-*` 或 `claude-*`。
- 作为 Agent 模块，我希望统一拿到 OpenAI 风格的响应结构，减少不同 provider 差异处理。
- 作为知识库模块，我希望未来 embedding 也能接入同一体系，但 V1 先把 chat 跑通。

### 7.3 运行时场景

- 作为系统，我希望当公有主模型失败时，先在 LiteLLM Proxy 内部 fallback，而不是应用内再重复切换一次同级公有模型。
- 作为系统，我希望当整个公有模型链路失败时，必要时可以切换到应用侧配置的私有模型兜底。

## 8. 范围定义

### 8.1 V1 必做范围

- 统一 Chat 模型调用接口。
- 支持非流式和流式输出。
- 支持 OpenAI 兼容的 messages 请求格式。
- 支持基础工具调用参数透传。
- 支持逻辑模型到网关模型名的映射。
- 支持租户、应用、场景级路由规则。
- 支持公有模型通过 LiteLLM Proxy 调用。
- 支持少量私有模型直连例外。
- 支持统一错误归一化。
- 支持 token、cost、latency、trace_id 记录。
- 支持基础 fallback 机制。

### 8.2 V1 预留但不强制实现

- embeddings 接口
- rerank 接口
- responses API
- 模型管理后台 UI
- 自动调优路由
- 灰度发布控制台

## 9. 职责边界

### 9.1 `app/modules/model_center/`

职责：

- 维护模型目录与元数据。
- 维护逻辑模型别名。
- 维护租户/应用/环境的模型可见性。
- 维护路由策略和 fallback 策略。
- 维护模型启停状态和基础配额规则。

非职责：

- 不直接调用 LiteLLM Proxy。
- 不直接调用 OpenAI、Anthropic 等 provider。

### 9.2 `app/runtime/llm/`

职责：

- 接收统一调用请求。
- 解析逻辑模型和上下文策略。
- 决定本次请求走 `litellm_proxy` 还是 `direct`。
- 调用对应 adapter。
- 统一做请求封装、响应标准化、错误归一化、日志归档。

非职责：

- 不维护第二套公有模型级路由器。
- 不在公有模型链路里重复做同级 fallback。

### 9.3 `app/integrations/model_providers/`

职责：

- 对接 LiteLLM Proxy。
- 对接例外直连 provider 或私有模型服务。
- 屏蔽 HTTP/SDK 协议差异。

### 9.4 `app/observability/`

职责：

- 记录 trace、latency、token、cost、error。
- 提供模型质量与稳定性分析数据。

## 10. 核心产品规则

### 10.1 单一主网关规则

公有模型默认只能通过 LiteLLM Proxy 访问。  
除非模型被标记为“直连例外”，否则应用侧不得直接访问公网 provider。

### 10.2 单一主要 fallback 执行层规则

同一条请求链路里，只允许一个主要的跨模型 fallback 执行层。

V1 规则如下：

- 公有模型链路：LiteLLM Proxy 负责模型级 fallback。
- 应用侧：仅允许做通道级 fallback，例如“公有链路整体失败后切到私有兜底模型”。

### 10.3 逻辑模型稳定规则

业务代码只允许引用逻辑模型，不允许引用 provider 原始模型名。  
例如业务代码可以写：

- `chat_default`
- `chat_summary`
- `private_sensitive`

不允许在业务代码里写：

- `gpt-4o`
- `claude-sonnet-*`
- `gemini-*`

### 10.4 配置下沉边界规则

应用侧配置只保存：

- LiteLLM Proxy Base URL
- LiteLLM Proxy API Key
- 少量直连例外模型配置

公网 provider 的 API Key、Base URL、deployment 等优先保存在 LiteLLM Proxy 配置和 Secret 中。

## 11. 功能需求

### 11.1 统一调用入口

系统必须提供统一内部调用接口，例如：

```python
invoke_chat(request: LLMInvokeRequest) -> LLMInvokeResult
stream_chat(request: LLMInvokeRequest) -> Iterator[LLMStreamChunk]
```

该接口对上层屏蔽以下差异：

- provider 请求结构差异
- 认证方式差异
- 响应结构差异
- 错误类型差异

### 11.2 逻辑模型解析

系统必须支持以下解析过程：

1. 根据 `tenant_id`、`app_id`、`scene`、`task_type`、`capability_tags` 选择逻辑模型。
2. 根据逻辑模型配置选择调用通道。
3. 如果通道为 `litellm_proxy`，解析成 LiteLLM Proxy 模型名。
4. 如果通道为 `direct`，解析成具体直连目标和 adapter。

### 11.3 路由规则

V1 支持以下规则维度：

- 按任务类型路由
- 按应用路由
- 按租户路由
- 按环境路由
- 按能力标签路由

V1 不支持：

- 基于实时学习的动态路由
- 基于在线效果分的自动调优

### 11.4 Fallback 规则

必须支持以下失败条件：

- 超时
- 429/限流
- 5xx 或 provider 不可用
- 输出不可解析
- 响应结构不满足契约

V1 fallback 策略：

- 第 1 层：LiteLLM Proxy 内部公有模型 fallback。
- 第 2 层：应用侧通道级 fallback 到私有兜底模型。

约束：

- 不能在 LiteLLM Proxy 和应用两边同时配同级公有模型 fallback。
- 必须限制最大 fallback 次数。
- 必须记录 fallback hop。

### 11.5 统一错误模型

系统内部至少归一化以下错误类型：

- `authentication_error`
- `permission_error`
- `rate_limit_error`
- `timeout_error`
- `provider_unavailable`
- `bad_response_error`
- `validation_error`
- `unknown_error`

### 11.6 观测与成本

每次调用至少记录以下字段：

- `trace_id`
- `tenant_id`
- `app_id`
- `scene`
- `logical_model`
- `final_channel`
- `final_provider`
- `final_model`
- `latency_ms`
- `prompt_tokens`
- `completion_tokens`
- `total_tokens`
- `cost`
- `status`
- `error_code`
- `fallback_count`

如果 LiteLLM Proxy 已输出 usage/cost，应用侧重点做业务归因，不重复实现底层计费。

## 12. 接口约定

### 12.1 内部请求对象

```python
from pydantic import BaseModel, Field

class LLMInvokeRequest(BaseModel):
    tenant_id: str
    app_id: str
    user_id: str | None = None
    scene: str
    task_type: str
    logical_model: str | None = None
    capability_tags: list[str] = Field(default_factory=list)
    stream: bool = False
    messages: list[dict]
    tools: list[dict] | None = None
    response_format: dict | None = None
    temperature: float | None = None
    max_tokens: int | None = None
    timeout_ms: int | None = None
    metadata: dict = Field(default_factory=dict)
```

说明：

- `logical_model` 可为空，表示由策略自动选择。
- `scene` 是路由的重要输入，例如 `chat`, `summary`, `code`, `sensitive_qa`。
- `metadata` 用于透传观测信息和审计上下文。

### 12.2 内部响应对象

```python
from pydantic import BaseModel, Field

class LLMInvokeResult(BaseModel):
    trace_id: str
    logical_model: str
    final_channel: str
    final_provider: str
    final_model: str
    content: str | None = None
    tool_calls: list[dict] | None = None
    finish_reason: str | None = None
    usage: dict = Field(default_factory=dict)
    cost: float | None = None
    latency_ms: int
    fallback_hops: list[dict] = Field(default_factory=list)
    raw_response: dict | None = None
```

### 12.3 LiteLLM Proxy 适配器约定

适配器职责：

- 用 OpenAI 兼容格式调用 LiteLLM Proxy。
- 设置统一 `base_url` 和 `api_key`。
- 支持非流式和流式请求。
- 读取 usage、model、finish_reason 等关键信息。
- 把 LiteLLM Proxy 错误映射到内部错误模型。

### 12.4 直连适配器约定

适配器职责：

- 只服务于少量例外链路。
- 保持与统一请求/响应模型兼容。
- 屏蔽私有模型服务协议差异。

## 13. 运行流程

### 13.1 标准请求链路

1. 上层模块发起 `LLMInvokeRequest`。
2. 治理层完成鉴权、租户识别、审计上下文注入。
3. `model_center` 根据租户、应用、场景、任务类型解析逻辑模型和可用通道。
4. `runtime/llm` 将逻辑模型解析为 LiteLLM Proxy 模型名，或解析为直连例外目标。
5. 若走公有模型链路，则由 LiteLLM Proxy 执行 provider 调用、路由和内部 fallback。
6. 若公有链路整体失败且配置允许，则 `runtime/llm` 切换到私有兜底模型。
7. `runtime/llm` 对响应做标准化、输出校验和错误归一化。
8. `observability` 记录调用明细。
9. 上层模块拿到统一结果。

### 13.2 Fallback 链路示例

```text
chat_default
  -> LiteLLM Proxy: public-chat-default
      -> OpenAI 主模型
      -> Anthropic 备用模型
  -> 应用侧兜底: private_sensitive_backup
```

这条链路的含义是：

- LiteLLM Proxy 负责公有模型组内部切换。
- 应用侧只在整条公有链路失败后，再切私有兜底。

## 14. 配置设计

### 14.1 应用侧环境变量

建议在 `.env.example` 中补充以下配置：

```env
# LiteLLM Proxy
MODEL_GATEWAY_BASE_URL=http://litellm-proxy:4000
MODEL_GATEWAY_API_KEY=change-me

# Gateway runtime
MODEL_GATEWAY_TIMEOUT_MS=60000
MODEL_GATEWAY_ENABLE_PUBLIC_PROXY=true
MODEL_GATEWAY_ENABLE_DIRECT_FALLBACK=true

# Direct exception models
PRIVATE_LLM_BASE_URL=
PRIVATE_LLM_API_KEY=
PRIVATE_LLM_MODEL=
```

### 14.2 LiteLLM Proxy 配置原则

LiteLLM Proxy 侧负责：

- 上游 provider 凭据管理
- 公有模型名暴露
- 公有模型内部路由
- 公有模型内部 fallback
- 基础 usage/cost 数据输出

应用侧不负责：

- 维护所有公网 provider 的 API Key
- 维护所有 provider 的原始 deployment 配置

LiteLLM Proxy 配置示意：

```yaml
model_list:
  - model_name: public-chat-default
    litellm_params:
      model: openai/<public-primary-model>
      api_key: os.environ/OPENAI_API_KEY

  - model_name: public-chat-backup
    litellm_params:
      model: anthropic/<public-backup-model>
      api_key: os.environ/ANTHROPIC_API_KEY

litellm_settings:
  master_key: os.environ/LITELLM_MASTER_KEY
  database_url: os.environ/LITELLM_DATABASE_URL
```

说明：

- 应用侧统一调用 `public-chat-default` 这类网关模型名，不直接调用上游 provider 模型名。
- 公有模型组内的具体 retry / fallback / routing 配置由 LiteLLM Proxy 侧维护。
- 不同 LiteLLM 版本的 router / fallback 配置细节可能不同，落地时应以当前官方文档为准。

### 14.3 模型配置对象建议

```yaml
logical_model: chat_default
channel: litellm_proxy
gateway_model_name: public-chat-default
task_type: chat
capability_tags:
  - tool-calling
tenant_scope:
  - tenant_a
app_scope:
  - app_chat
fallback_policy: public_chat_fallback
enabled: true
```

## 16. 模块实现建议

### 16.1 目录落位

```text
app/
├─ modules/
│  └─ model_center/
│     ├─ services/
│     │  ├─ model_catalog_service.py
│     │  ├─ model_route_service.py
│     │  └─ model_policy_service.py
│     └─ repositories/
├─ runtime/
│  └─ llm/
│     ├─ schemas.py
│     ├─ gateway_service.py
│     ├─ model_resolver.py
│     ├─ response_normalizer.py
│     └─ error_mapper.py
├─ integrations/
│  └─ model_providers/
│     ├─ litellm_proxy_adapter.py
│     ├─ private_llm_adapter.py
│     └─ base.py
└─ observability/
   ├─ tracing/
   └─ metrics/
```

### 16.2 核心组件说明

`ModelResolver`

- 输入：`LLMInvokeRequest`
- 输出：解析后的逻辑模型、调用通道、目标 adapter、目标模型名

`GatewayService`

- 编排完整调用流程
- 负责选择 adapter
- 负责通道级 fallback
- 负责记录调用结果

`LiteLLMProxyAdapter`

- 负责 OpenAI 兼容请求
- 负责流式与非流式调用
- 负责从 Proxy 响应中抽取 usage/model/error

`ResponseNormalizer`

- 统一 content/tool_calls/usage/finish_reason
- 保证上层只消费统一格式

`ErrorMapper`

- 把 LiteLLM Proxy 或直连 provider 错误映射成内部标准错误

## 17. 分阶段实现建议

### 17.1 第一阶段

- 接入 LiteLLM Proxy
- 打通统一 chat 调用
- 打通非流式和流式请求
- 打通逻辑模型到网关模型映射
- 打通 token/cost/latency 记录

### 17.2 第二阶段

- 加入租户和应用级模型策略
- 加入通道级 fallback 到私有模型
- 加入模型启停与权限控制
- 加入基础管理 API

### 17.3 第三阶段

- 支持 embedding
- 支持 rerank
- 支持更细粒度预算与成本归因
- 支持观测结果回流优化策略

## 18. 验收标准

### 18.1 功能验收

- 上层业务不再直接引用任何公网 provider SDK。
- 同一个内部接口可以调用 LiteLLM Proxy 和私有直连模型。
- 公有模型默认只通过 LiteLLM Proxy 调用。
- 请求失败时，fallback 行为符合配置且不会双重执行。
- 上层拿到的始终是统一响应结构。

### 18.2 数据验收

- 每次请求都能记录 trace、latency、usage、cost、error。
- 能按租户、应用、逻辑模型维度检索调用记录。
- 能区分最终走的是 `litellm_proxy` 还是 `direct`。

### 18.3 架构验收

- `model_center` 不直接发起 provider 调用。
- `runtime/llm` 不维护第二套公有模型级路由器。
- `integrations/model_providers` 仅承担协议适配职责。

## 19. 风险与约束

### 19.1 风险

- 如果应用侧和 LiteLLM Proxy 同时维护公有模型 fallback，链路会变得不可控。
- 如果业务代码继续直接写 provider 模型名，逻辑模型体系会失效。
- 如果公网 provider 凭据继续保存在应用侧，模型网关边界会被破坏。
- 如果 V1 一开始就做 embedding、rerank、多模态，会显著拉高落地复杂度。

### 19.2 约束

- V1 必须优先保证边界清晰，而不是功能全。
- V1 必须先保证 chat 场景稳定。
- V1 必须让 LiteLLM Proxy 成为公有模型的唯一主网关。

## 20. 外部依赖说明

LiteLLM 官方文档明确将 Proxy Server 定位为集中式 LLM Gateway，并支持：

- OpenAI 兼容输入输出格式
- 多 provider 统一接入
- 路由 / retry / fallback
- 成本跟踪与预算管理

因此，本项目把 LiteLLM Proxy 用作公有模型主网关是成立的，但应用侧仍需保留：

- 逻辑模型目录
- 策略路由
- 通道分发
- 例外直连
- 业务归因观测
