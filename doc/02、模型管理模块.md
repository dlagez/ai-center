# 模型管理模块说明

## 1. 模块定位

模型管理模块是 AI 中台里的基础能力模块，目标不是“再包一层模型调用”，而是把多模型接入、路由、fallback、成本控制、租户隔离和运行策略统一收口。

在本项目的单体服务架构中，它承担三件事：

- 向上为 Prompt、Agent、知识库、应用发布等模块提供统一的模型使用入口。
- 向下屏蔽不同模型供应商之间的接口差异、认证方式差异和计费差异。
- 在运行时控制“该用哪个模型、失败后切到哪个模型、成本如何统计、不同租户如何隔离”。

它对应 [01、架构.md](doc/01%E3%80%81%E6%9E%B6%E6%9E%84.md) 中的“模型与工具抽象层”以及“A. 模型管理中心”。

## 2. 适用场景

这个模块适合当前 AI 中台/企业项目的典型场景：

- 需要同时接入多个 provider。
- 需要统一配置不同模型的访问方式。
- 需要根据租户、应用、任务类型做模型路由。
- 需要配置 fallback，避免单一模型失败导致整体服务不可用。
- 需要统一统计 token、费用、调用次数和失败率。
- 后续要接 LangGraph 工作流和 LangSmith 评测体系。

如果系统只有一个模型、没有租户隔离、也不需要成本治理，那么这个模块可以极简实现；但对于中台场景，它通常必须存在。

## 3. 当前方案结论

当前阶段建议采用“统一模型接入层 + LiteLLM Proxy 为主入口 + 保留私有模型直连能力”的方案。

结论拆开说是：

- 公有大模型优先通过 LiteLLM Proxy 接入。
- 私有模型、本地模型、vLLM、自建推理服务保留直连适配能力。
- 上层业务只依赖统一模型接口，不直接依赖具体 provider SDK。

这样设计的原因是：

- LiteLLM Proxy 适合承接多 provider 的统一入口和统一鉴权出口。
- 公有模型的跨 provider 路由、fallback 和底层 usage/cost 统计优先收敛到 LiteLLM Proxy 侧。
- 应用侧只保留统一模型接口、策略决策和少量调用编排，早期可以显著降低多模型接入复杂度。
- 后续接 LangGraph、LangSmith 时，上层工作流不需要感知底层 provider 变化。

这里要注意两点：

- “使用 LiteLLM Proxy 作为模型网关”不等于所有模型都必须走 LiteLLM Proxy。对于企业内部私有模型、对数据合规要求很高的模型，仍然应该支持直连或内网代理。
- 同一条请求链路里，跨模型路由和跨模型 fallback 只应有一个主要执行层。默认情况下，这个执行层应是 LiteLLM Proxy，而不是应用和网关同时执行。

## 4. 在单体服务中的职责边界

模型管理模块不等于整个模型运行时。按当前目录设计，它和其他目录的边界如下：

- `app/modules/model_center/`
  负责模型目录、元数据、逻辑模型别名、路由策略、fallback 策略、租户/应用可见性、配额和成本规则管理，输出声明式策略，不直接执行 provider 调用。
- `app/runtime/llm/`
  负责统一调用编排、请求封装、响应标准化、超时/重试控制，以及“走 LiteLLM Proxy 还是走直连例外链路”的执行分发。默认不再维护第二套公有模型路由器。
- `app/integrations/model_providers/`
  负责对接 LiteLLM Proxy，以及私有模型、内网模型、例外直连 provider 的具体协议。
- `app/observability/`
  负责追踪、指标、评测和反馈闭环，不直接决定模型怎么选，但会为模型治理提供数据。

也就是说：

- `model_center` 管“目录、策略和配置”。
- `runtime/llm` 管“统一调用编排和执行分发”。
- `integrations/model_providers` 管“网关和例外直连适配”。
- 默认由 LiteLLM Proxy 执行公有模型的跨 provider 路由与 fallback。

## 5. 核心能力说明

### 5.1 多模型接入

模型管理模块首先要解决“模型来源很多”的问题。

这里要区分两层概念：

- 上游 provider：OpenAI、Anthropic、Gemini、Azure OpenAI 等。
- 接入通道：LiteLLM Proxy、私有模型直连、内网代理等。

需要支持的模型来源与通道组合至少包括：

- 通过 LiteLLM Proxy 接入的公有模型：OpenAI、Anthropic、Gemini、Azure OpenAI 等。
- 通过 LiteLLM Proxy 或内网代理暴露的标准化私有模型。
- 保留直连例外的本地模型、企业私有模型、vLLM 或自建推理服务。

统一接入后，上层不再关心底层差异，只关心“我要 chat、embedding、rerank 还是 structured output”。

### 5.2 模型注册与元数据管理

模型不能只存一个名字，还需要维护元数据。建议最少管理以下信息：

- 模型唯一标识 `model_id`
- 显示名称
- 上游 provider 名称
- 接入通道：`litellm_proxy` / `direct` / `internal_proxy`
- 网关模型名或逻辑别名（例如 LiteLLM Proxy 暴露的 `model_name`）
- 任务类型：chat、embedding、rerank
- 能力标签：vision、tool-calling、JSON 输出、长上下文、多模态
- 最大上下文长度
- 默认超时
- 单次重试次数
- 单价或计费规则
- 是否启用
- 是否默认
- 适用环境：dev、test、prod
- 适用租户或适用应用范围

这些元数据是后续做路由、成本控制和灰度发布的基础。

### 5.3 模型路由

路由是模型管理模块的核心能力之一。

在使用 LiteLLM Proxy 的前提下，路由要分成两层理解：

- `model_center` 决定“逻辑上允许用哪些模型、哪个模型组优先”。
- LiteLLM Proxy 负责执行公有模型的具体路由与跨 provider 切换。

路由不只是“选一个模型”，而是根据上下文决定当前请求最合适的模型。常见策略包括：

- 按任务类型路由
  例如对话走通用模型，摘要走低成本模型，代码任务走代码模型。
- 按能力路由
  例如需要工具调用时只选支持 tool calling 的模型。
- 按租户路由
  某些租户只能使用私有模型，某些租户允许使用公有模型。
- 按应用路由
  不同应用绑定不同默认模型。
- 按环境路由
  开发环境使用便宜模型，生产环境使用稳定模型。
- 按成本路由
  在满足效果要求的前提下优先选择成本更低的模型。
- 按健康状态路由
  某个模型错误率升高时临时降权或摘除。

初期建议先做“固定路由 + 规则路由”，不要一开始就做复杂的动态学习路由。对于公有模型，优先把执行路由收敛到 LiteLLM Proxy 的模型别名或模型组中，而不是在应用内枚举大量 provider 细节。

### 5.4 Fallback

fallback 是保证可用性的关键能力。

在 LiteLLM Proxy 为主网关的前提下，fallback 也要明确执行边界：

- 公有模型链路：默认由 LiteLLM Proxy 执行跨模型 fallback。
- 私有模型直连例外链路：由 `app/runtime/llm/` 执行必要的 fallback。

当主模型出现以下情况时，应自动切换到备用模型：

- 调用超时
- provider 接口报错
- 返回内容不可解析
- 命中限流
- 当前模型不可用

fallback 需要明确三件事：

- 切换条件是什么
- 备用链路顺序是什么
- 是否允许跨 provider 切换

一个简单的例子是：

`业务逻辑模型别名 -> LiteLLM Proxy 内同组备用模型 -> 应用侧直连的企业私有模型（最终兜底）`

同时要注意：

- 不能在 LiteLLM Proxy 和应用两侧同时做同级 fallback 或同级重试。
- 不能无限 fallback，必须限制次数。
- fallback 结果要被记录，用于后续稳定性分析。
- 对需要强一致输出格式的场景，要校验 fallback 后是否仍满足输出契约。

### 5.5 配额与成本

中台项目不能只关心“能不能调用”，还要关心“调用成本是否可控”。

模型管理模块至少要支持以下治理能力：

- 按租户统计费用
- 按应用统计费用
- 按模型统计费用
- 按 provider 统计费用
- 统计 prompt token / completion token / request 数
- 设置调用配额和预算阈值
- 超额后的限制策略

成本统计不一定要求第一版就做到精确财务级别，但至少要满足“可观测、可预警、可追溯”。

如果 LiteLLM Proxy 已输出基础 usage/spend 数据，应用侧重点应放在租户、应用、场景维度的业务归因，而不是重复实现一套底层计费逻辑。

### 5.6 私有模型 / 公有模型切换

企业场景里，这个能力很重要。

常见诉求包括：

- 某些高敏感数据只能走私有模型。
- 某些低敏感或通用问答可以走公有模型。
- 某些租户购买了专属模型能力。
- 某些场景为了成本考虑优先走公有模型。

因此模型管理模块需要支持：

- 基于数据等级的模型选择
- 基于租户策略的模型选择
- 基于应用策略的模型选择
- 基于环境配置的切换

推荐的方向不是“全局只配一个默认模型”，而是“按策略决定是否走公有/私有模型”。

### 5.7 观测与评测数据回流

模型管理模块虽然不直接承担 LangSmith 的全部能力，但必须消费观测数据。

例如：

- 哪个模型平均延迟更高
- 哪个模型成本更高
- 哪个模型 fallback 频率更高
- 哪个模型在某类任务上失败率更高
- 哪个模型在人工标注评测中得分更高

这些数据会直接影响路由和模型下线决策。

## 6. 典型调用流程

一个典型请求从上到下大致是这样流转的：

1. 上层模块发起模型调用请求，例如 Chat、Agent 节点、RAG 生成阶段。
2. 治理层完成鉴权、租户识别、配额校验和审计上下文注入。
3. 模型管理模块根据任务类型、租户、应用、能力要求选择逻辑模型、允许的调用通道和策略规则。
4. `runtime/llm` 将逻辑模型解析为 LiteLLM Proxy 的模型别名，或解析为直连例外目标，并发起统一调用。
5. 默认情况下，由 LiteLLM Proxy 执行公有模型的具体路由、fallback 与上游 provider 调用。
6. 对直连例外链路，由具体 provider 适配器调用私有模型或内网推理服务。
7. `runtime/llm` 完成响应标准化、输出契约校验，并把 token、费用、耗时、错误信息送入观测与评测体系。
8. 上层模块收到统一格式的结果，而不是 provider 原始响应。

这条链路意味着模型管理模块本质上是中台里的“模型决策层”。

## 7. 与其他模块的协同关系

### 7.1 与 Prompt 管理中心

Prompt 管理中心决定“怎么提问”，模型管理模块决定“用哪个模型去回答”。

两者之间应该解耦：

- Prompt 模块不硬编码具体 provider。
- 模型模块不存具体 Prompt 模板内容。

### 7.2 与 Agent 工作流中心

LangGraph 编排的每个节点都可能需要不同模型。

因此 Agent 工作流中心应调用模型管理模块提供的统一接口，而不是直接在节点内写死：

- `gpt-4`
- `claude`
- `gemini`

否则后续路由、fallback、评测都无法集中治理。

### 7.3 与知识库中心

知识库中心不仅需要 chat 模型，还需要：

- embedding 模型
- reranker 模型
- 生成模型

这些模型也应该统一纳入模型管理范围，而不是由知识库模块单独维护配置。

### 7.4 与监控评测中心

监控评测中心输出：

- trace
- latency
- token
- cost
- failure case
- 评测结果

模型管理模块消费这些数据，用于：

- 调整路由
- 下线模型
- 变更默认模型
- 优化 fallback 策略

## 8. 单体项目中的建议目录映射

结合当前目录结构，模型管理相关代码后续建议主要落在以下位置：

```text
app/
├─ modules/
│  └─ model_center/               # 模型中心业务逻辑与管理接口
├─ runtime/
│  └─ llm/                        # 模型运行时、统一调用、网关/直连链路分发
├─ integrations/
│  └─ model_providers/            # LiteLLM Proxy 与直连例外 provider 适配器
├─ observability/
│  ├─ tracing/
│  ├─ metrics/
│  └─ evals/
└─ governance/
   ├─ quota/
   ├─ billing/
   └─ routing/
```

建议职责如下：

- `modules/model_center`
  管理模型配置、模型目录、逻辑模型别名、路由规则、fallback 策略、启停状态。
- `runtime/llm`
  提供统一调用入口、请求封装、响应标准化、超时/重试控制，以及网关/直连链路分发。
- `integrations/model_providers`
  封装 LiteLLM Proxy 以及直连例外 provider 的 SDK 或 HTTP 协议差异。

## 9. 推荐的配置维度

后续真正实现时，模型管理模块建议至少支持以下配置维度：

- 网关配置（LiteLLM Proxy Base URL / API Key）
- 直连例外配置（私有模型 Base URL / 内网认证信息）
- 模型目录
- 逻辑模型别名与网关模型名映射
- 默认模型
- 按场景的模型映射
- fallback 链
- 超时与重试参数
- token 与成本归因规则
- 租户级启用策略
- 应用级启用策略
- 环境隔离配置

如果后续使用 LiteLLM Proxy，应用侧 `.env` 应优先收敛成“统一模型网关配置 + 少量直连例外配置”的形式；公网 provider 的个性化配置原则上放在 LiteLLM Proxy 的配置或 Secret 中，而不是默认下沉到应用侧。

## 10. 分阶段落地建议

### 第一阶段

先把最核心能力做起来：

- LiteLLM Proxy 接入
- 公有 chat 模型统一走 LiteLLM Proxy
- 固定路由
- 简单 fallback（只在一侧执行）
- token / cost / latency 记录

### 第二阶段

再补治理能力：

- 多租户模型权限
- 公有/私有模型切换
- 模型启停管理
- 路由规则管理
- 应用级模型绑定

### 第三阶段

最后补优化能力：

- LangSmith 评测回流
- 基于观测结果的模型调优
- 更细粒度的成本治理
- embedding / rerank / 多模态统一纳管

## 11. 本模块的设计原则

最后把原则明确下来：

- 上层模块不直接依赖具体 provider。
- 模型选择逻辑集中管理，不分散到业务代码里。
- 公有模型和私有模型都纳入统一治理体系。
- 成本、稳定性、能力边界必须可观测。
- 路由和 fallback 先做规则化，再逐步演进。
- 同一条请求链路只允许一个主要的跨模型路由 / fallback 执行层，默认是 LiteLLM Proxy。
- 应用侧默认不保存全部公网 provider 凭据，优先收敛到模型网关配置。
- 单体服务阶段优先保证实现简单、边界清晰、后续可扩展。

## 12. 当前结论摘要

基于当前项目阶段，模型管理模块的建议结论是：

- 这是 AI 中台必须独立出来的核心模块。
- 当前最合适的落地方向是“统一模型入口 + LiteLLM Proxy 作为公有模型主网关 + 私有模型保留例外直连”。
- 该模块不仅负责模型接入，还负责路由、fallback、成本治理和模型策略管理。
- 它会成为后续 LangGraph 工作流、知识库检索、Prompt 发布和 LangSmith 评测的共同底座。
